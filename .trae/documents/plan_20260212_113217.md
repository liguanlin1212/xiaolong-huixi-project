# 任务1.2：NPU 推理集成实施计划

## 目标
根据迭代流程文档和强约束实现说明，集成真实 NPU 推理能力，实现端侧文本分类的高效执行。

## 实施步骤

### 1. 准备模型目录结构
- **创建 `ai/models/` 目录**：用于存放 ONNX 格式的文本分类模型
- **准备轻量级文本分类模型**：选择适合端侧推理的小型模型

### 2. 实现 ONNX Runtime 集成
- **安装必要的依赖**：ONNX Runtime 及其 NPU 扩展（如果可用）
- **修改 `npu_runner.py`**：
  - 添加模型加载逻辑
  - 实现基于 ONNX Runtime 的推理
  - 保持接口契约不变（只返回 `label` 和 `confidence`）
  - 确保向后兼容性

### 3. 模型转换与优化
- **转换模型为 ONNX 格式**：确保模型结构适合文本分类任务
- **模型量化**：将模型量化为 INT8 以提高 NPU 性能
- **模型优化**：针对端侧推理进行必要的优化

### 4. 推理逻辑实现
- **文本预处理**：实现文本到模型输入的转换
- **推理执行**：使用 ONNX Runtime 执行模型推理
- **结果后处理**：将模型输出转换为标准的分类结果
- **错误处理**：添加适当的错误处理机制

### 5. 性能优化
- **批处理推理**：实现批处理以提高吞吐量
- **缓存优化**：优化模型加载和推理缓存
- **内存管理**：确保内存使用高效

### 6. 测试与验证
- **功能测试**：验证文本分类功能是否正常
- **性能测试**：测试推理速度和资源使用情况
- **准确性测试**：验证分类准确性
- **兼容性测试**：确保在不同环境下的兼容性

## 技术要点

### 模型选择
- 使用轻量级文本分类模型（如小型 BERT 或 CNN）
- 确保模型大小适合端侧部署
- 优先选择支持 ONNX 格式的模型

### 推理引擎
- 优先使用 ONNX Runtime with NPU 支持
- 提供 CPU 推理作为备用方案
- 确保推理不依赖网络连接

### 性能优化
- 模型量化为 INT8
- 批处理推理
- 内存使用优化

### 接口设计
- 保持与现有 `classify_text` 函数接口一致
- 确保返回值严格符合强约束文档要求
- 提供清晰的错误处理机制

## 预期结果
- 成功集成 ONNX Runtime 推理能力
- 模型加载和推理性能满足端侧要求
- 文本分类功能正常工作
- 系统能在本地完成文本分类，不依赖网络
- 与现有代码无缝集成
